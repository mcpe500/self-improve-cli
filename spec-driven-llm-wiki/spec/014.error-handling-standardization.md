# Spec 014: Error Handling Standardization

## Task/Prompt

Standardize error handling across all modules. Currently 4 inconsistent patterns are used. Adopt a single Result<T,E> pattern for library code and exceptions for truly exceptional cases.

## Tujuan

- One error handling pattern across all modules
- No silent catch blocks
- No `process.stderr.write` in library code
- Proper input validation at module boundaries
- Predictable error propagation

## Current State (4 Patterns)

| Pattern | Used By | Example |
|---------|---------|---------|
| Throw exceptions | `validateProfile`, `askToolPermission`, `ensureAllowed` | `throw new Error('...')` |
| Return `{ok: false, error}` | `normalizeToolExecution`, tool handlers | `{ok: false, error: '...'}` |
| Return nullable/error object | `reviewQuestion` | `{approved: false, reason: '...'}` |
| Write to stderr/stdout | MCP init, daemon loop | `process.stderr.write('...')` |

## Target State

**Library code** (src/ modules): Throw exceptions for invalid inputs, return `{ok, result/error}` for operations that can fail predictably.

**CLI code** (bin/): Catch exceptions, print user-friendly messages, exit with proper code.

**No silent catches.** Every catch block must either re-throw, log, or return an error.

## Logic Changes

### Rule 1: Input validation throws

```javascript
function doSomething(arg) {
  if (!arg || typeof arg !== 'string') {
    throw new TypeError('arg must be a non-empty string');
  }
  // ...
}
```

### Rule 2: Operations that can fail return Result

```javascript
// Result type: { ok: true, result: T } | { ok: false, error: string }
async function readFileMaybe(path) {
  try {
    const content = await fs.readFile(path, 'utf8');
    return { ok: true, result: content };
  } catch (err) {
    return { ok: false, error: err.message };
  }
}
```

### Rule 3: No silent catch blocks

```javascript
// BEFORE (silent):
try { ... } catch {}

// AFTER (logged):
try { ... } catch (err) {
  process.stderr.write(`warning: ${err.message}\n`);
}
```

### Rule 4: Library code never writes to stdout/stderr

Only CLI entrypoint and chat loop should write to stdout/stderr. Library functions return errors or throw.
